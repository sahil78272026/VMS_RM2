# RM2 Visitor Management System (VMS) — System Architecture & Documentation

This document outlines the complete architecture of the RM2 VMS platform, including the tech stack, external services, and how all the different components talk to each other.

---

## 1. High-Level Architecture

The platform is a Monorepo split into three distinct applications:
1. **Backend** (Python / FastAPI)
2. **Mobile App** (React Native / Expo)
3. **Web Kiosk / Landing Page** (React / Vite)

### Interaction Flow
- A **Visitor** approaches the gate and scans the physical QR code (generated by the Guard App) with their personal phone.
- This opens the **Web Kiosk**, where they take a selfie and submit a check-in request.
- The **Backend** receives the request, uploads the selfie to Cloud Storage, and triggers a **Push Notification** to the **Resident's Mobile App**.
- The **Resident** approves the visitor on their phone.
- The **Guard's Mobile App** automatically updates (via Server-Sent Events / SSE or polling) to show the approval, and the guard marks the entry.

---

## 2. Component Tech Stack

### A. Backend (`/backend`)
The core server handling all business logic, database connections, and external API integrations.
*   **Framework:** FastAPI (Python 3.12.3)
*   **Server:** Gunicorn with Uvicorn async workers
*   **Database ORM:** SQLAlchemy
*   **Migrations:** Alembic
*   **Authentication:** JWT (JSON Web Tokens)
*   **Hosting:** Render (Web Service)

### B. Mobile App (`/mobile`)
The unified mobile application used by Residents, Guards, and Admins. The UI changes dynamically based on the user's role.
*   **Framework:** React Native with Expo (SDK 54)
*   **Navigation:** React Navigation (Native Stack)
*   **HTTP Client:** Axios (with interceptors for automatic JWT refreshing)
*   **Local Storage:** AsyncStorage (for session persistence)
*   **QR Code:** `react-native-qrcode-svg` (For guard check-in display)
*   **Hosting / Builds:** Expo Application Services (EAS)

### C. Web Kiosk / Landing Page (`/landing-page`)
A lightweight, mobile-responsive web app designed specifically for visitor self-check-in.
*   **Framework:** React 19 + Vite
*   **Styling:** Custom CSS (Glassmorphism design aesthetic)
*   **Hardware Access:** WebRTC / HTML5 Camera API (for live selfie capture)
*   **Hosting:** Netlify

## 3. External Cloud Services & Integrations

### 1. Supabase (Database & Storage)
*   **PostgreSQL Database:** Hosts all relational data (Users, Flats, Gates, Visitor Logs, Notifications). Connected to the Backend via standard `DATABASE_URL`.
*   **S3 Object Storage:** Hosts visitor selfies.
    *   *Workflow:* Web Kiosk captures a Base64 selfie → Sends to Backend → Backend uses `boto3` to convert to binary and stream to the Supabase `rm2-bucket` → Saves the public URL in the database.

### 2. Push Notifications Pipeline (Expo & Firebase)
We utilize a managed proxy architecture for Push Notifications. Rather than configuring raw Firebase credentials in the backend, the backend communicates entirely with Expo, which manages the underlying native delivery systems.

*   **Firebase Cloud Messaging (FCM):** Under the hood, Android devices require Firebase to wake up the phone and display a notification. When the `.apk` is built via EAS, Expo automatically bundles the necessary FCM native code into the Android app.
*   **Apple Push Notification Service (APNs):** For iOS devices, Expo automatically interfaces with APNs.
*   **The Workflow:**
    1.  **App Load:** When a resident opens the mobile app, the `expo-notifications` library generates a unique `Expo Push Token` and saves it to the Backend database.
    2.  **Event Trigger:** A visitor checks in at the Web Kiosk.
    3.  **Backend Dispatch:** The FastAPI Backend sends a simple JSON payload containing the Resident's `Expo Push Token` to the **Expo Push API** (`https://exp.host/--/api/v2/push/send`).
    4.  **Expo Routing:** Expo's cloud servers identify the device type from the token. If it's an Android device, Expo securely forwards the payload to **Firebase (FCM)**.
    5.  **Final Delivery:** Firebase wakes up the resident's phone and displays the system alert.

### 3. Render (Backend Hosting)
*   Hosts the FastAPI application.
*   Runs the `alembic upgrade head` migration script automatically on every deployment before spinning up the Uvicorn worker.

### 4. Netlify (Frontend Hosting)
*   Hosts the static Vite React build for the Web Kiosk.

## 4. Environment Variables Mapping

The system relies heavily on `.env` files to route traffic between the local development environment and production.

### Backend (`backend/.env`)
*   `DATABASE_URL`: Connection string to Supabase PostgreSQL.
*   `APP_ENV`: Controls logging (`DEV` vs `PROD`).
*   `SECRET_KEY` & `JWT_SECRET_KEY`: Cryptographic keys for secure user sessions.
*   `SUPABASE_S3_*`: Credentials required by `boto3` to upload images to the bucket.

### Mobile App (`mobile/.env` & `eas.json`)
*   `EXPO_PUBLIC_API_URL`: Points the mobile app to the Render backend (e.g., `https://vms-rm2-1.onrender.com/api/v1`).
*   `EXPO_PUBLIC_FRONTEND_URL`: Injected into the Guard's QR code generator to point visitors to the Web Kiosk (e.g., `https://vmsrm2.netlify.app`).

### Web Kiosk (`landing-page/.env`)
*   `VITE_API_URL`: Points the web app to the Render backend.

## 5. Security & Authentication

1.  **JWT Rotation:** The mobile app receives an `access_token` (expires in 24h) and a `refresh_token` (expires in 30 days). Axios interceptors automatically fetch a new access token seamlessly when it expires.
2.  **Role-Based Access Control (RBAC):** Every API route in FastAPI is protected by role-checking middleware (e.g., `@require_role(["admin", "guard"])`).
3.  **Kiosk Lockdown:** The Web Kiosk strictly uses the front-facing camera API. It completely bypasses the device's file picker, ensuring visitors must take a live selfie rather than uploading a fake photo.
